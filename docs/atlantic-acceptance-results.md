## Scenarios A-E (status)

The original `scripts/atlantic-acceptance/scenario-{a..e}.sh` were
shell stubs (`echo` + `exit 0`). They are now thin bash wrappers
that exec the matching `scenario-{a..e}.mjs` Node.js script in
`scripts/atlantic-acceptance/`. Each Node.js script exercises the
implementation in-process (no API server, no live blockchain):

| Scenario | Status | Script | Notes |
|----------|--------|--------|-------|
| A — Successful multi-agent job   | **PASS** (in-process) | `scenario-a.mjs` | Drives a 3-task job through the `Orchestrator`; asserts all tasks end `VERIFIED` and `totalSpent` matches the per-task cost. |
| B — Failed worker reassignment   | **PASS** (in-process) | `scenario-b.mjs` | Exercises the bounded-retry path (first two attempts fail, third succeeds) and the persistent-failure path (worker always throws, task ends `FAILED`). |
| C — Verifier disagreement        | **PASS** (in-process) | `scenario-c.mjs` | Verifier returns `ok = false` for t3; t1 and t2 are `VERIFIED`, t3 is `FAILED`. |
| D — Budget rejection             | **PASS** (in-process) | `scenario-d.mjs` | A spec whose per-task budgets sum to more than the job budget is rejected by the workflow compiler (`ValidationError` with `code: "budget_overflow"`); the orchestrator marks every task `FAILED` with `totalSpent == 0`. |
| E — Risky-target rejection       | **PASS** (in-process) | `scenario-e.mjs` | `StaticGoplusClient` returns `verdict: "risky"` for a denylisted address; the end-to-end job aborts with `goplus_risky_target`. |
| F — Final receipt verification   | **PASS** (on-chain) | `scenario-f.mjs` -> `on-chain-roundtrip.mjs` | The on-chain roundtrip above proves the contract returns the same roots that were submitted. |

Sample output of every in-process scenario is recorded in
`docs/local-acceptance-results.md` alongside the vitest
unit/integration tests. The Node.js scripts exit 0 on success
and non-zero on failure so they can be wired into a future CI
gate.

The shell wrappers can also be re-implemented as end-to-end API
exercises (e.g. `curl http://localhost:8787/jobs -d @spec.json
...` followed by a `getReceipt(jobId)` read from the chain) once
the API server is deployed to Alibaba Cloud; the in-process
scripts make the API deploy an enhancement rather than a
prerequisite for the acceptance proof.
